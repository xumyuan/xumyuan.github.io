UE 材质到 Shader 处理机制解析 ​

基于当前工程源码整理，涉及路径均相对仓库根目录。本文聚焦"一个材质资产如何变成 GPU 上可执行的 Shader"，与 Docs/UE_Shader机制详解.md（讲 FShaderType/FShader/编译缓存底层）互补，可对照阅读。

一、总览：材质 → Shader 的四个阶段 ​

材质系统跨越 游戏线程 UObject 与 渲染侧对象 两大世界，从"节点图"到"GPU 字节码"要经过四个阶段：

  1. 对象建模：UMaterial/UMaterialInstance（游戏线程资产）通过 FMaterialRenderProxy 暴露给渲染器；每种 (FeatureLevel, QualityLevel) 对应一个渲染侧 FMaterialResource（继承自 FMaterial）。
  2. 翻译（Translate）：FHLSLMaterialTranslator 遍历材质表达式节点图，把每个材质属性（BaseColor/Metallic/Normal…）编译成 HLSL 代码块，填进模板 MaterialTemplate.ush，生成一份虚拟源码 /Engine/Generated/Material.ush。
  3. 编译与缓存（Compile & Cache）：FMaterialShaderMap 针对材质 × 每个 FVertexFactoryType × 每个 FShaderType 派发编译作业，结果存入 DDC 并在内存中按 FMaterialShaderMapId 去重。
  4. 运行期取用（Draw）：MeshPassProcessor 用材质 + 顶点工厂类型从 FMaterialShaderMap 取出具体 FShader，再从 FMaterialRenderProxy 求值参数、构建 FMeshDrawCommand 并提交 RHI。

二、对象建模：从 UObject 到渲染侧 FMaterial ​

2.1 游戏线程类层次 ​

  • UMaterialInterface 是抽象基类，定义 GetMaterial()、GetRenderProxy()、GetMaterialResource() 等纯虚接口：
cpp
// Engine/Source/Runtime/Engine/Classes/Materials/MaterialInterface.h (L331-361)
	virtual class UMaterial* GetMaterial() PURE_VIRTUAL(UMaterialInterface::GetMaterial,return NULL;);
	...
	virtual class FMaterialRenderProxy* GetRenderProxy() const PURE_VIRTUAL(UMaterialInterface::GetRenderProxy,return NULL;);
  • UMaterial 是真正拥有节点图的资产。它持有一个"渲染代理" DefaultMaterialInstance 和一组按质量/特性等级切分的渲染资源 MaterialResources：
cpp
// Engine/Source/Runtime/Engine/Classes/Materials/Material.h (L1411-1437)
	class FDefaultMaterialInstance* DefaultMaterialInstance;
	...
	/** 
	 * Material resources used for rendering this material.
	 * There need to be as many entries in this array as can be used simultaneously for rendering.  
	 */
	TArray<FMaterialResource*> MaterialResources;
  • UMaterialInstance（MIC/MID 的基类）不含节点图，只覆写参数；它持有父材质 Parent 与自己的渲染代理 Resource。仅当存在静态开关参数（bHasStaticPermutationResource）时，才会拥有独立的 StaticPermutationMaterialResources（需要重新编译出新的 Shader 排列），否则直接复用父材质的 Shader：
cpp
// Engine/Source/Runtime/Engine/Private/Materials/MaterialInstance.cpp (L2164-2177)
FMaterialResource* UMaterialInstance::GetMaterialResource(ERHIFeatureLevel::Type InFeatureLevel, EMaterialQualityLevel::Type QualityLevel, uint8 RenderProfileID)
{
	if (bHasStaticPermutationResource)
	{
		return FindMaterialResource(StaticPermutationMaterialResources, InFeatureLevel, QualityLevel, RenderProfileID, true);
	}
	return Parent ? Parent->GetMaterialResource(InFeatureLevel, QualityLevel, RenderProfileID) : nullptr;
}

结论：MIC/MID 若只改标量/向量/贴图参数，不会触发 Shader 重新编译——它们与父材质共享同一份 FMaterialShaderMap，参数值仅在运行期通过 FMaterialInstanceResource 的参数数组注入到 Material UniformBuffer。只有**静态开关（StaticSwitch）**等改变代码结构的参数才会产生新的 Shader 排列。

2.2 渲染侧：FMaterial / FMaterialResource ​

FMaterial 是抽象基类，源码注释明确其三重职责：

cpp
// Engine/Source/Runtime/Engine/Public/MaterialShared.h (L1500-1505)
 * FMaterial serves 3 intertwined purposes:
 *   Represents a material to the material compilation process, and provides hooks for extensibility (CompileProperty, etc)
 *   Represents a material to the renderer, with functions to access material properties
 *   Stores a cached shader map, and other transient output from a compile, which is necessary with async shader compiling

它持有两个关键的 ShaderMap 指针——游戏线程一份、渲染线程一份，跨线程更新：

cpp
// Engine/Source/Runtime/Engine/Public/MaterialShared.h (L2161-2173)
	/** 
	 * Game thread tracked shader map...
	 * During an async compile, this will be NULL...
	 */
	TRefCountPtr<FMaterialShaderMap> GameThreadShaderMap;

	/** 
	 * Shader map for this material resource which is accessible by the rendering thread. 
	 */
	TRefCountPtr<FMaterialShaderMap> RenderingThreadShaderMap;

SetGameThreadShaderMap 在游戏线程赋值后，通过 ENQUEUE_RENDER_COMMAND 把新指针同步到渲染线程，这是保证线程安全的关键约定：

cpp
// Engine/Source/Runtime/Engine/Public/MaterialShared.h (L1929-1940)
	void SetGameThreadShaderMap(FMaterialShaderMap* InMaterialShaderMap)
	{
		checkSlow(IsInGameThread() || IsInAsyncLoadingThread());
		GameThreadShaderMap = InMaterialShaderMap;
		TRefCountPtr<FMaterialShaderMap> ShaderMap = GameThreadShaderMap;
		TRefCountPtr<FMaterial> Material = this;
		ENQUEUE_RENDER_COMMAND(SetGameThreadShaderMap)([Material = MoveTemp(Material), ShaderMap = MoveTemp(ShaderMap)](FRHICommandListImmediate& RHICmdList) mutable
		{
			Material->RenderingThreadShaderMap = MoveTemp(ShaderMap);
		});
	}

FMaterialResource 是 FMaterial 面向 UMaterial/UMaterialInstance 的具体实现，回指两个 UObject：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/MaterialShared.cpp (L2053-2056)
UMaterialInterface* FMaterialResource::GetMaterialInterface() const
{
	return MaterialInstance ? (UMaterialInterface*)MaterialInstance : (UMaterialInterface*)Material;
}

2.3 缓存入口：CacheResourceShadersForRendering ​

游戏线程侧真正触发编译的入口。它按需为当前激活的 FeatureLevel/QualityLevel 查找或创建 FMaterialResource，然后调用 CacheShadersForResources：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/Material.cpp (L3543-3584)
void UMaterial::CacheResourceShadersForRendering(bool bRegenerateId)
{
	...
	const EMaterialQualityLevel::Type ActiveQualityLevel = GetCachedScalabilityCVars().MaterialQualityLevel;
	...
	while (FeatureLevelsToCompile != 0)
	{
		const ERHIFeatureLevel::Type FeatureLevel = (ERHIFeatureLevel::Type)FBitSet::GetAndClearNextBit(FeatureLevelsToCompile);
		FMaterialResource* CurrentResource = FindOrCreateMaterialResource(MaterialResources, this, nullptr, FeatureLevel, ActiveQualityLevel, ActiveProfileID, bAllowDefaultQuality);
		ResourcesToCache.Add(CurrentResource);
		CacheShadersForResources(ShaderPlatform, ResourcesToCache);
	}

三、翻译阶段：节点图 → HLSL 源码 ​

3.1 FHLSLMaterialTranslator ​

翻译器实现了 FMaterialCompiler 接口，是节点图遍历与 HLSL 代码生成的核心。它以"代码块"（FShaderCodeChunk）为基本单位按频率（VS/PS 等）分桶存放：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/HLSLMaterialTranslator.h (L195-279)
class FHLSLMaterialTranslator : public FMaterialCompiler
{
protected:
	EShaderFrequency ShaderFrequency;
	EMaterialProperty MaterialProperty;
	...
	TArray<FShaderCodeChunk> SharedPropertyCodeChunks[SF_NumFrequencies];
	/** Code chunk definitions corresponding to each of the material inputs */
	FString TranslatedCodeChunkDefinitions[CompiledMP_MAX];
	/** Code chunks corresponding to each of the material inputs */
	FString TranslatedCodeChunks[CompiledMP_MAX];
	FString MaterialTemplate;

这里的翻译结果是一堆材质属性的getter函数，供后续shader复用

3.2 逐属性编译，Normal 优先 ​

Translate() 会对每个材质属性（EMaterialProperty，如 MP_BaseColor/MP_Metallic/MP_Normal）调用 CompilePropertyAndSetMaterialProperty。Normal 必须最先编译，因为其它属性可能依赖法线结果：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/HLSLMaterialTranslator.cpp (L676-702)
		// Normal must always be compiled first; this will ensure its chunk calculations are the first to be added
		{
			check(SharedPropertyCodeChunks[NormalShaderFrequency].Num() == 0);
			Chunk[MP_Normal]					= Material->CompilePropertyAndSetMaterialProperty(MP_Normal					,this);
			NormalCodeChunkEnd = SharedPropertyCodeChunks[NormalShaderFrequency].Num();
		}
		...
		Chunk[MP_BaseColor]						= Material->CompilePropertyAndSetMaterialProperty(MP_BaseColor				,this);
		Chunk[MP_Metallic]						= Material->CompilePropertyAndSetMaterialProperty(MP_Metallic				,this);
		Chunk[MP_Roughness]						= Material->CompilePropertyAndSetMaterialProperty(MP_Roughness				,this);

注：本工程在此基础上扩展了额外的自定义属性（如 MP_CustomData2~7、MP_Porosity、MP_CustomGI 等），机制完全一致，只是属性枚举更长。

3.3 节点图遍历与去重：CallExpression ​

每个材质属性对应一条 FExpressionInput，CompileWithDefault 沿连线走到上游节点。遍历入口 UMaterial::CompilePropertyEx 把每个 MP_* 映射到材质对应的输入引脚：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/Material.cpp (L6495-6497)
	case MP_BaseColor: return BaseColor.CompileWithDefault(Compiler, Property);
	case MP_Normal:    return Normal.CompileWithDefault(Compiler, Property);
	case MP_Metallic:  return Metallic.CompileWithDefault(Compiler, Property);

翻译器的 CallExpression 是中央调度器：它先查 ExpressionCodeMap 是否已编译过该节点（跨属性复用同一子图），再调用节点虚函数 Compile()，最后缓存结果：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/HLSLMaterialTranslator.cpp (L3172)
	int32 Result = ExpressionKey.Expression->Compile(Compiler, ExpressionKey.OutputIndex);

3.4 节点如何发射代码：AddCodeChunk / AddUniformExpression ​

每个 UMaterialExpression::Compile() 先递归编译输入，再调用 FMaterialCompiler 的算子（Add/Mul/TextureSample…），返回的是代码块索引（相当于 SSA 变量 ID），而非 HLSL 字符串。例如加法节点：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/MaterialExpressions.cpp (L3549-3557)
int32 UMaterialExpressionAdd::Compile(class FMaterialCompiler* Compiler, int32 OutputIndex)
{
	int32 Arg1 = A.GetTracedInput().Expression ? A.Compile(Compiler) : Compiler->Constant(ConstA);
	int32 Arg2 = B.GetTracedInput().Expression ? B.Compile(Compiler) : Compiler->Constant(ConstB);
	return Compiler->Add(Arg1, Arg2);
}

翻译器实现里，AddCodeChunk 会为每个表达式生成一个局部变量 LocalN，并对代码串做哈希以去重：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/HLSLMaterialTranslator.cpp (L2443-2453)
	const FString SymbolName = CreateSymbolName(TEXT("Local"));
	const FString LocalVariableDefinition =
		FString("	") + HLSLTypeString(Type) + TEXT(" ") + SymbolName
		+ TEXT(" = ") + FormattedCode + TEXT(";") + LINE_TERMINATOR;
	new(*CurrentScopeChunks) FShaderCodeChunk(Hash, *LocalVariableDefinition, SymbolName, Type, false);

两类代码块的区别：

  • AddCodeChunk：随像素/顶点逐次求值的运行时表达式（如两个贴图相乘）。
  • AddUniformExpression：可在 CPU 侧折叠或来自参数的值（标量/向量/贴图参数、常量），运行期存进 Material UniformBuffer，通过 Material.ScalarExpressions[] 等在 Shader 中访问。这正是 MID 改参数无需重编 的底层原因。

3.5 填模板：MaterialTemplate.ush + FLazyPrintf ​

MaterialTemplate.ush 是带一堆 %s 占位符的 HLSL 骨架，例如两个核心结构体和逐属性 getter 函数体：

hlsl
// Engine/Shaders/Private/MaterialTemplate.ush (L207-218)
struct FMaterialAttributes
{
%s
};
struct FPixelMaterialInputs
{
%s
};

GetMaterialShaderCode() 用 FLazyPrintf 从左到右依次把翻译出的代码串塞进每个 %s。每个属性 getter 的函数体形如 [局部变量定义] return [表达式];：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/HLSLMaterialTranslator.cpp (L2025-2029)
FString FHLSLMaterialTranslator::GenerateFunctionCode(uint32 Index) const
{
	return TranslatedCodeChunkDefinitions[Index] + TEXT("	return ") + TranslatedCodeChunks[Index] + TEXT(";");
}

3.6 交棒给编译系统：虚拟包含 /Engine/Generated/Material.ush ​

生成的整份 HLSL 不落盘，而是作为"虚拟包含文件"注册进编译环境。各 Mesh/Material Shader 的 .usf 只需 #include "/Engine/Generated/Material.ush"：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/MaterialShared.cpp (L2844-2863)
	FHLSLMaterialTranslator MaterialTranslator(this, NewCompilationOutput, StaticParameterSet, Platform,GetQualityLevel(), ShaderMapId.FeatureLevel, TargetPlatform);
	bSuccess = MaterialTranslator.Translate();

	if(bSuccess)
	{
		TRefCountPtr<FShaderCompilerEnvironment> MaterialEnvironment = new FShaderCompilerEnvironment();
		...
		MaterialTranslator.GetMaterialEnvironment(Platform, *MaterialEnvironment);
		const FString MaterialShaderCode = MaterialTranslator.GetMaterialShaderCode();
		...
		MaterialEnvironment->IncludeVirtualPathToContentsMap.Add(TEXT("/Engine/Generated/Material.ush"), MaterialShaderCode);

		// Compile the shaders for the material.
		NewShaderMap->Compile(this, ShaderMapId, MaterialEnvironment, NewCompilationOutput, Platform, bSynchronousCompile);

GetMaterialEnvironment() 还会写入一系列 #define MATERIAL_* 宏（BlendMode、ShadingModel、各种特性开关），供模板 #if 分支使用。

四、编译与缓存：FMaterialShaderMap ​

4.1 ShaderMap 的结构 ​

FMaterialShaderMap 保存一个材质排列的全部编译产物。其内容分两层：材质级 Shader（无顶点工厂）存在基类容器里，Mesh 级 Shader 按 FVertexFactoryType 分组存进 OrderedMeshShaderMaps：

cpp
// Engine/Source/Runtime/Engine/Public/MaterialShared.h (L1014-1069)
class FMaterialShaderMapContent : public FShaderMapContent
{
	...
	LAYOUT_FIELD(TMemoryImageArray<TMemoryImagePtr<FMeshMaterialShaderMap>>, OrderedMeshShaderMaps);
	LAYOUT_FIELD(FMaterialCompilationOutput, MaterialCompilationOutput);
	LAYOUT_FIELD(FSHAHash, ShaderContentHash);
};

因此运行期取 Shader 是两级键：先用 FVertexFactoryType 找到 FMeshMaterialShaderMap，再用 FShaderType 哈希名 + PermutationId 找到具体 FShader。

4.2 缓存键：FMaterialShaderMapId ​

FMaterialShaderMapId 决定"两个材质能不能共用一份编译结果"。它涵盖材质 GUID、质量/特性等级、静态开关参数集、以及所有依赖的 Shader 类型与顶点工厂类型的源码哈希：

cpp
// Engine/Source/Runtime/Engine/Public/MaterialShared.h (L760-836)
class FMaterialShaderMapId
{
	FSHAHash CookedShaderMapIdHash;
	FGuid BaseMaterialId;
	EMaterialQualityLevel::Type QualityLevel;
	ERHIFeatureLevel::Type FeatureLevel;
	...
	TArray<FStaticSwitchParameter> StaticSwitchParameters;
	...
	TArray<FShaderTypeDependency> ShaderTypeDependencies;
	TArray<FVertexFactoryTypeDependency> VertexFactoryTypeDependencies;
	FSHAHash TextureReferencesHash;
	FSHAHash BasePropertyOverridesHash;
};

其中每个依赖项都带上了源文件哈希，这样改了 .usf 源码后，SourceHash 变化 → Id 变化 → 自动失效旧缓存并重编：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/MaterialShader.cpp (L880-908)
	Dependency.ShaderTypeName = ShaderTypes[i]->GetHashedName();
	Dependency.SourceHash = ShaderTypes[i]->GetSourceHash(ShaderPlatform);
	...
	Dependency.VertexFactoryTypeName = VFTypes[i]->GetHashedName();
	Dependency.VFSourceHash = VFTypes[i]->GetSourceHash(ShaderPlatform);

4.3 三级缓存查找 ​

FMaterial::CacheShaders 依次尝试：内存去重表 → DDC → 从头编译：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/MaterialShared.cpp (L2667-2678)
	TRefCountPtr<FMaterialShaderMap> ShaderMap = FMaterialShaderMap::FindId(ShaderMapId, Platform);
	if (!ShaderMap && !FPlatformProperties::RequiresCookedData())
	{
		FMaterialShaderMap::LoadFromDerivedDataCache(this, ShaderMapId, Platform, TargetPlatform, LoadedShaderMap);
	}

内存去重表 GIdToMaterialShaderMap 按平台维护 Id → ShaderMap，多个相同材质共享同一份：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/MaterialShader.cpp (L47-49)
FCriticalSection FMaterialShaderMap::GIdToMaterialShaderMapCS;
TMap<FMaterialShaderMapId,FMaterialShaderMap*> FMaterialShaderMap::GIdToMaterialShaderMap[SP_NumPlatforms];

DDC 键以 MATSM 为前缀，由 AppendKeyString 拼出完整指纹：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/MaterialShader.cpp (L151-159)
	ShaderMapId.AppendKeyString(ShaderMapKeyString);
	return FDerivedDataCacheInterface::BuildCacheKey(TEXT("MATSM"), MATERIALSHADERMAP_DERIVEDDATA_VER, *ShaderMapKeyString);

4.4 派发编译作业：Compile ​

FMaterialShaderMap::Compile 先取"布局"（FMaterialShaderMapLayout，缓存了哪些 Shader×VF 组合需要编译），然后为每个顶点工厂建一个 FMeshMaterialShaderMap 并派发作业，最后把作业交给 GShaderCompilingManager：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/MaterialShader.cpp (L1345-1616)
void FMaterialShaderMap::Compile(...)
{
	...
	const FMaterialShaderMapLayout& Layout = AcquireMaterialShaderMapLayout(InPlatform, MaterialParameters);

	// Per VF: 建 FMeshMaterialShaderMap，派发 mesh 作业
	for (const FMeshMaterialShaderMapLayout& MeshLayout : Layout.MeshShaderMaps) {
		MeshShaderMap = new FMeshMaterialShaderMap(InPlatform, MeshLayout.VertexFactoryType);
		MeshShaderMap->BeginCompile(CompilingId, InShaderMapId, Material, MeshLayout, ...);
	}
	// Per FMaterialShaderType
	for (const FShaderLayoutEntry& Shader : Layout.Shaders) {
		if (!Material->ShouldCache(...)) continue;
		ShaderType->BeginCompileShader(...);
	}
	...
	GShaderCompilingManager->AddJobs(NewJobs, bSynchronousCompile || !Material->IsPersistent(), ...);

哪些排列真正需要编译由两层 ShouldCompilePermutation 门控：布局层（哪些 ShaderType×VF×Permutation 组合存在）+ 每材质层（FMaterial::ShouldCache）。FMeshMaterialShaderType::BeginCompileShader 会让顶点工厂注入自己的编译宏后调用 GlobalBeginCompileShader：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/MeshMaterialShader.cpp (L31-95)
FShaderCompileJob* FMeshMaterialShaderType::BeginCompileShader(...)
{
	FShaderCompileJob* NewJob = new FShaderCompileJob(ShaderMapId, VertexFactoryType, this, PermutationId);
	NewJob->Input.SharedEnvironment = MaterialEnvironment;
	VertexFactoryType->ModifyCompilationEnvironment(..., ShaderEnvironment);
	...
	::GlobalBeginCompileShader(Material->GetFriendlyName(), VertexFactoryType, this, ShaderPipeline, ...);
	return NewJob;
}

4.5 异步编译与"默认材质"兜底 ​

若是异步编译，GameThreadShaderMap 会被置空，编译完成前渲染回退到默认材质：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/MaterialShared.cpp (L2875-2877)
		// Async compile, use NULL so that rendering will fall back to the default material.
		OutShaderMap = nullptr;

编译作业跑完后，ShaderCompiler.cpp 把编译好的 Shader 装入 ShaderMap，置 bCompilationFinalized，回写 DDC，并把 GameThreadShaderMap 指回完成的图：

cpp
// Engine/Source/Runtime/Engine/Private/ShaderCompiler/ShaderCompiler.cpp (L2708-2871)
	if (bShaderMapComplete) {
		ShaderMap->bCompiledSuccessfully = bSuccess;
		ShaderMap->ProcessCompilationResults(...);  // 置 bCompilationFinalized
		...
		Material->GameThreadShaderMap = CompletedShaderMap;
		Material->NotifyCompilationFinished();
	}

五、运行期取用：从材质到 Draw Call ​

5.1 取材质与兜底 ​

MeshPassProcessor::AddMeshBatch 通过 FMaterialRenderProxy::GetMaterialWithFallback 拿到可用的 FMaterial。若该材质的 RenderingThreadShaderMap 尚未就绪，就返回默认材质代理：

cpp
// Engine/Source/Runtime/Renderer/Private/BasePassRendering.cpp (L1496-1504)
void FBasePassMeshProcessor::AddMeshBatch(const FMeshBatch& RESTRICT MeshBatch, ...)
{
	const FMaterialRenderProxy* FallbackMaterialRenderProxyPtr = nullptr;
	const FMaterial& Material = MeshBatch.MaterialRenderProxy->GetMaterialWithFallback(FeatureLevel, FallbackMaterialRenderProxyPtr);
	const FMaterialRenderProxy& MaterialRenderProxy = FallbackMaterialRenderProxyPtr ? *FallbackMaterialRenderProxyPtr : *MeshBatch.MaterialRenderProxy;

FDefaultMaterialInstance 的兜底实现——只有 ShaderMap 已 finalize 才返回自身，否则退回域默认材质：

cpp
// Engine/Source/Runtime/Engine/Private/Materials/Material.cpp (L404-422)
	const FMaterialResource* MaterialResource = Material->GetMaterialResource(InFeatureLevel, QualityLevel);
	if (MaterialResource && MaterialResource->GetRenderingThreadShaderMap())
	{
		checkSlow(MaterialResource->GetRenderingThreadShaderMap()->IsCompilationFinalized());
		return *MaterialResource;
	}
	check(!Material->IsDefaultMaterial());
	OutFallbackMaterialRenderProxy = &GetFallbackRenderProxy();
	return OutFallbackMaterialRenderProxy->GetMaterialWithFallback(InFeatureLevel, OutFallbackMaterialRenderProxy);

5.2 用顶点工厂类型取具体 Shader ​

六、Shader 字节码的存储与分发：Shader Library ​

前面的链路里，FShader 本身不存字节码，只存一个 ResourceIndex；真正的字节码由 FShaderMapResource 持有。那字节码从哪来、怎么落盘、运行时怎么加载？这就是 Shader Code Library（Shader 代码库） 负责的下游环节。本工程（定制引擎）在此基础上做了扩展，默认库名为 Global / 项目名。

6.1 三层数据模型 ​

类型职责
FShaderMapResourceCode真正持有字节码 ShaderEntries[].Code + 每个 shader 的哈希 + 汇总 ResourceHash
FShaderMapResource运行期 RHI 持有者，RHIShaders[ShaderIndex] 数组；FShader::ResourceIndex 索引它
FShaderMapResource_InlineCode字节码内联在材质资产里的实现
FShaderMapResource_SharedCode字节码来自 Shader Library 的实现（仅存在于 .cpp）

6.2 序列化的岔路：内联 vs 共享库 ​

这里是否使用共享库的标志是填入了资产文件的，所以加载的时候会自动判别是加载inline的shader代码（字节码uexp文件），还是用hash码去资产库找对应的字节码

FMaterialShaderMap::Serialize 最终走到 FShaderMapBase::Serialize。是否入库由 bShareCode 决定——Cook 时开启了"Share Material Shader Code"就入库，否则字节码内联进资产：

cpp
// Engine/Source/Runtime/RenderCore/Private/ShaderMap.cpp (L198-219)
		bool bShareCode = false;
#if WITH_EDITOR
		bShareCode = !bInlineShaderCode && FShaderCodeLibrary::IsEnabled() && Ar.IsCooking();
#endif // WITH_EDITOR
		Ar << bShareCode;
		...
		if (bShareCode)
		{
			FSHAHash ResourceHash = Code->ResourceHash;
			Ar << ResourceHash;
			FShaderCodeLibrary::AddShaderCode(GetShaderPlatform(), Code, GetAssociatedAssets(), GetRenderProfileName());
		}
		else
		{
			Code->Serialize(Ar, bLoadedByCookedMaterial);
		}
  • 入库：材质资产里只写一个 ResourceHash，字节码交给 FShaderCodeLibrary::AddShaderCode 按哈希去重后累积到库里。
  • 内联：字节码原样写进资产（本工程定制引擎还支持 bShaderCodeZstdCompress 的 Lz4 压缩）。

这里序列化和反序列化的代码都在一个函数中处理，通过Ar.IsSaving()来控制

对应的加载侧：bShareCode 时用 ResourceHash 去库里 LoadResource，否则新建 FShaderMapResource_InlineCode：

cpp
// Engine/Source/Runtime/RenderCore/Private/ShaderMap.cpp (L281-304)
		bool bShareCode = false;
		Ar << bShareCode;
		if (bShareCode)
		{
			FSHAHash ResourceHash;
			Ar << ResourceHash;
			Resource = FShaderCodeLibrary::LoadResource(ResourceHash, &Ar);
			...
		}
		else
		{
			Code = new FShaderMapResourceCode();
			Code->Serialize(Ar, bLoadedByCookedMaterial, bShaderCodeZstdCompress);
			Resource = new FShaderMapResource_InlineCode(GetShaderPlatform(), Code);
		}

意义：默认材质不能依赖库（启动早期库还没开），因此 ShouldInlineShaderCode() 会强制它们内联；普通材质则共享库，避免同一份字节码在成千上万个材质资产里重复存储。

6.6 与 PSO 缓存（FShaderPipelineCache）的关系 ​

Shader Library 只管字节码；FShaderPipelineCache 管PSO（管线状态对象）预编译，且强依赖 Shader Library 开启。链路是：Cook 写 .scl.csv 稳定键 → 运行/开发期记录 PSO 得到 .stablepc.csv → Commandlet 合并成 .upipelinecache → 运行时按稳定键解析回 FSHAHash，从已打开的 Shader Library 取字节码预建 PSO。库开/关会通知 PSO 缓存：

cpp
// Engine/Source/Runtime/RenderCore/Private/ShaderCodeLibrary.cpp (L2551-2552)
				FShaderPipelineCache::ShaderLibraryStateChanged(FShaderPipelineCache::Opened, ShaderPlatform, Name);

一句话定位：Shader Library 不改变"材质怎么变成 Shader"，它只决定"编译好的字节码怎么去重存盘、运行时怎么按需加载"——是链路最末端的存储/分发层。它在编辑器 PIE（未 Cook、未开该选项）下通常不生效，字节码走内联；打包后才启用。

七、关键结论速记 ​

问题源码结论
一个材质对应几份 Shader？每个 (FeatureLevel, QualityLevel, RenderProfile) 一个 FMaterialResource，各自一份 FMaterialShaderMap；每份内又按"顶点工厂类型 × ShaderType × Permutation"展开成很多 FShader。
MID 改参数会重编 Shader 吗？不会。标量/向量/贴图参数走 UniformExpression → Material UniformBuffer；只有静态开关等结构性参数才产生新排列。
节点图怎么变成 HLSL？FHLSLMaterialTranslator 遍历图，每个节点 Compile() 发射代码块，最终 FLazyPrintf 填入 MaterialTemplate.ush，生成虚拟包含 /Engine/Generated/Material.ush。
编译结果怎么复用？内存 GIdToMaterialShaderMap 去重 + DDC（键前缀 MATSM）；键 FMaterialShaderMapId 含源码哈希，源码变更自动失效。
编译没完成时怎么渲染？GameThreadShaderMap 为空，GetMaterialWithFallback 回退到域默认材质。
Draw 时怎么找到 Shader？两级键：FVertexFactoryType → FMeshMaterialShaderMap，再 FShaderType + PermutationId → FShader。
字节码存哪、Shader Library 是什么？FShader 只存 ResourceIndex，字节码在 FShaderMapResource。Cook 且开启 bShareMaterialShaderCode 时字节码按哈希去重存进 .ushaderbytecode（Shader Library），资产只留 ResourceHash；否则内联进资产。运行期 CreateRHIShader 从库或内联码建 FRHIShader。
Shader Library 属于哪一环？链路最末端的存储/分发层，不影响"材质→Shader"的翻译与编译；PIE/未打包下通常走内联。
