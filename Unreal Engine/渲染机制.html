UE4 渲染管线笔记 ​

1. 概念速览 ​

概念粒度职责对应源码
UPrimitiveComponentComponent游戏线程的渲染组件PrimitiveComponent.h
FPrimitiveSceneProxyComponent（渲染副本）渲染线程的数据镜像，剥离UObjectPrimitiveSceneProxy.h
FPrimitiveSceneInfoComponent场景端管理信息（Bounds、可见性等）PrimitiveSceneInfo.h
FScene全局场景中所有Primitive的注册表Scene.h
FSceneRenderer帧一帧的渲染调度者SceneVisibility.cpp
FMeshElementCollector帧收集所有动态/静态MeshBatchMeshElementCollector.h
FMeshBatchDraw Call一组顶点数据 + 材质 + 渲染状态MeshBatch.h
FMeshBatchElementInstance 组MeshBatch内的子绘制单元，通常为1MeshBatch.h
FMeshPassMaskDraw Call位掩码，标记走哪些PassMeshPassProcessor.h

粒度层级关系 ​

Actor > Component (1:1 Primitive) > Section (1:N MeshBatch) > Triangle
  │              │                       │
  │      可见性/剔除单元           一次 Draw Call
  │      1个Primitive             1个材质Section
  └── 场景对象                   = 1个MeshBatch

2. 线程模型 ​

Game Thread                          Render Thread
─────────────                         ──────────────
UPrimitiveComponent  ──创建镜像──→   FPrimitiveSceneProxy
  - UObject引用                       - ❌ 不拷贝
  - 材质列表                           - 材质RenderProxy指针（拷贝）
  - Transform                          - Transform（拷贝值）
  - CastShadow                         - CastShadow（拷贝值）
  - 蓝图状态                           - ❌ 渲染不需要

Proxy.GetDynamicMeshElements()  ←── 渲染线程安全调用
  - 不访问任何 UObject
  - 只读 Proxy 上的缓存数据

核心原则：Proxy 创建时一次性拷贝，后续 Game Thread 修改通过异步消息同步。

3. 一帧渲染主流程 ​

代码位置：

  • Render() → DeferredShadingRenderer.cpp
  • InitViews() / ComputeViewVisibility() / GatherDynamicMeshElements() → SceneVisibility.cpp
  • OcclusionCull() → SceneVisibility.cpp（内部调用 SceneOcclusion.cpp 的硬件查询）

剔除在哪完成的？ ​

ComputeViewVisibility 内部依次执行视锥剔除 + 遮挡剔除，结果写入 View.PrimitiveVisibilityMap（位图）。这个位图派生出的 HasDynamicMeshElementsMasks 传给 GatherDynamicMeshElements — 这里只消费结果，不自己做剔除。

总体的剔除大概有下面几个过程：

  1. 视锥剔除，OBB级别的剔除
  2. 显示隐藏剔除
  3. showonly过滤
  4. 静态场景过滤
  5. 线框小物体剔除
  6. 遮挡剔除

OBB = Oriented Bounding Box（有向包围盒）。是紧贴物体跟着变换矩阵一同变换的包围盒

4. 动态网格收集 ​

4.1 GatherDynamicMeshElements 做了什么 ​

cpp
// 位置: SceneVisibility.cpp
void FSceneRenderer::GatherDynamicMeshElements(
    TArray<FViewInfo>& Views,
    const FScene* Scene,
    const FPrimitiveViewMasks& HasDynamicMeshElementsMasks,  // 上游剔除结果
    FMeshElementCollector& Collector)

对每个可见的 Primitive：

  1. 读 ViewMask：该 Primitive 对哪些 View 可见
  2. 调用 PrimitiveSceneInfo->Proxy->GetDynamicMeshElements() → 让 Proxy 自己往 Collector 里塞 MeshBatch
  3. 记录 DynamicMeshEndIndices：标记该 Primitive 的 MeshBatch 范围
  4. 调用 ComputeDynamicMeshRelevance()：给新加的 MeshBatch 打 Pass 标签

注意："Dynamic" 指网格顶点数据每帧需要重新生成（如SkeletalMesh蒙皮、粒子系统），不是"没被剔除"的意思。StaticMesh 的静态数据走另一条路径，不经过这里。

4.2 GetDynamicMeshElements（以 SkeletalMesh 为例） ​

CreateBaseMeshBatch 填充了什么：

cpp
Mesh.VertexFactory        = 蒙皮顶点工厂       // CPU蒙皮 or GPU蒙皮？
Mesh.MaterialRenderProxy  = 材质RenderProxy     // 用哪个材质？

BatchElement.FirstIndex    = Section.BaseIndex     // 索引缓冲起始
BatchElement.IndexBuffer   = LODData.IndexBuffer   // 指向哪段索引缓冲
BatchElement.NumPrimitives = Section.NumTriangles  // 三角形数量
BatchElement.PrimitiveUniformBuffer = Transform等  // 公共UBO

不拷贝顶点数据，只记录"去缓冲的哪个位置，读多少，用什么着色器"。

5. Pass 分类 ​

5.1 ComputeDynamicMeshRelevance 的作用 ​

决定"这个 MeshBatch 需要被哪些渲染 Pass 画"，同时累计每个 Pass 的元素计数。

5.2 两个维度的输出 ​

输出类型粒度用途
PassMask位掩码每个 MeshBatch这个 MeshBatch 要走哪些 Pass
View.NumVisibleDynamicMeshElements[Pass]计数器每个 View × 每个 Pass该 Pass 有多少个 Element（≈ Draw Call 数）

5.3 计数器的意义 ​

cpp
// 渲染 BasePass 前，已知一共 1500 个 DrawCall
VisibleMeshDrawCommands.Reserve(NumVisibleDynamicMeshElements[BasePass]);  // 精确预分配

for (每个 MeshBatch) {
    if (PassMask.Has(BasePass)) {
        VisibleMeshDrawCommands.Add(生成绘制命令);
    }
}
  • 预分配数组 → 避免运行时扩容
  • 某 Pass 计数为 0 → 整个 Pass 跳过
  • 一次性填充 Command Buffer → 减少 GPU 提交次数

6. 从 FMeshBatch 到 FMeshDrawCommand ​

上一节已经得到 DynamicMeshElements、每个 MeshBatch 的 PassMask，以及每个 Pass 的预计 Element 数量。接下来要做的事只有一个：为每个 Pass 把相关的 FMeshBatch 转成可排序、可提交的绘制命令列表。

6.1 总体调用链 ​

FSceneRenderer::SetupMeshPass（SceneRendering.cpp）遍历 EMeshPass，但只处理带 EMeshPassFlags::MainView 标志的 Pass。它通过 FPassProcessorManager 找到该 Pass 注册的创建函数，创建对应的 FMeshPassProcessor，然后调用：

cpp
View.ParallelMeshDrawCommandPasses[PassIndex].DispatchPassSetup(...);

Mobile BasePass 等少数 Pass 在这里会被跳过，改由 Mobile 渲染路径稍后单独派发；这属于调度特例，不影响下面的命令生成主流程。

6.2 FMeshPassProcessor：决定一个 MeshBatch 如何进入当前 Pass ​

FMeshPassProcessor 是 FMeshBatch → FMeshDrawCommand 的转换器。不同 Pass 使用不同派生类，但遵循相同模式：

  1. AddMeshBatch 判断该 Mesh 是否应该进入当前 Pass。
  2. 选择 Shader、材质策略和渲染状态。
  3. 调用公共的 BuildMeshDrawCommands 写出命令。

以 FBasePassMeshProcessor::AddMeshBatch 为例，它会：

  • 根据 BlendMode 区分不透明和不同类型的半透明 BasePass；
  • 检查 MainPass、MaterialDomain 等条件；
  • 选择 LightMapPolicy 和对应 Shader；
  • 计算 FillMode、CullMode；
  • 最终调用 BuildMeshDrawCommands(...)。

BuildMeshDrawCommands（MeshPassProcessor.inl）负责与具体 Pass 无关的装配工作：

  • 从 VertexFactory 获取 VertexDeclaration 和 VertexStreams；
  • 收集 Pass、材质及每个 FMeshBatchElement 的 ShaderBindings；
  • 组装 Shader、Blend、Rasterizer、DepthStencil 等 Minimal PSO 状态；
  • 遍历 BatchElementMask，通常为每个 FMeshBatchElement 生成一个 FMeshDrawCommand；
  • 同时创建指向它的 FVisibleMeshDrawCommand，加入本 Pass 的本帧列表。

因此，AddMeshBatch 负责“画不画、怎么画”，BuildMeshDrawCommands 负责“把决定编码成绘制命令”。

6.3 FMeshDrawCommandPassSetupTask：生成、排序和实例化合并 ​

DispatchPassSetup 本身不生成命令。它先填充 FMeshDrawCommandPassSetupTaskContext，并根据下面的上限在渲染线程预分配内存：

cpp
MaxNumDraws =
    InOutMeshDrawCommands.Num()
    + NumDynamicMeshElements
    + NumDynamicMeshCommandBuildRequestElements;

然后根据配置并行或同步执行 FMeshDrawCommandPassSetupTask。Task 的主流程是：

  1. GenerateDynamicMeshDrawCommands 遍历 DynamicMeshElements。
  2. 检查该 MeshBatch 的 PassRelevance，只把属于当前 Pass 的 Mesh 交给 Processor->AddMeshBatch。
  3. 处理需要本帧重新构建命令的 DynamicMeshCommandBuildRequests。
  4. ApplyViewOverridesToMeshDrawCommands 应用反转剔除、双面渲染、BasePass DepthStencil 等 View 级覆盖。
  5. 半透明 Pass 根据 View 和 PrimitiveBounds 更新深度 SortKey。
  6. 按 SortKey、StateBucketId 排序，使相同状态桶的命令相邻。
  7. 启用 GPUScene 时，建立 PrimitiveId 数据；旧 GPUScene 路径还会在 BuildMeshDrawCommandPrimitiveIdBuffer 中把相邻且 StateBucketId 相同的命令合并为一个实例化 Draw，并增加其 NumInstances。

FMeshDrawCommandInitResourcesTask 不负责上述内存预分配。它只在平台不能于工作线程按需创建 Shader 时，回到渲染线程对 Minimal PSO 中的 Shader 执行 LazilyInitShaders()。

6.4 三个核心数据结构 ​

FGraphicsMinimalPipelineStateInitializer

保存与 RenderTarget 无关的 PSO 状态，如 Shader、Blend、Rasterizer、DepthStencil 和 PrimitiveType。FinalizeCommand 会把它登记到 FGraphicsMinimalPipelineStateSet，并让 FMeshDrawCommand 只保存一个 CachedPipelineId，用于状态去重。

实际提交时，FMeshDrawCommand::SubmitDrawBegin 才把它转成完整的 FGraphicsPipelineStateInitializer，再通过当前 RHI CommandList 补齐 RT 信息：

cpp
FGraphicsPipelineStateInitializer GraphicsPSOInit =
    MeshPipelineState.AsGraphicsPipelineStateInitializer();
RHICmdList.ApplyCachedRenderTargets(GraphicsPSOInit);
SetGraphicsPipelineState(RHICmdList, GraphicsPSOInit, ...);

FMeshDrawCommand

真正的绘制命令数据，包含：

  • ShaderBindings；
  • VertexStreams 和 IndexBuffer；
  • CachedPipelineId；
  • FirstIndex、NumPrimitives、NumInstances；
  • BaseVertex / IndirectArgs；
  • StencilRef 等非 PSO 状态。

动态路径生成的命令本体存放在 FDynamicMeshDrawCommandStorage 中。

FVisibleMeshDrawCommand

本帧参与当前 Pass 的轻量包装，持有指向 FMeshDrawCommand 的指针，并额外保存 SortKey、StateBucketId、PrimitiveId、Fill/Cull Mode 等排序和实例化信息。

cpp
typedef TArray<FVisibleMeshDrawCommand, SceneRenderingAllocator>
    FMeshCommandOneFrameArray;

把二者分开后，排序时只移动较小的 FVisibleMeshDrawCommand，不用搬动包含大量资源绑定信息的 FMeshDrawCommand。
