DepthFade / Cast-as-Masked 阴影 Pass 崩溃问题记录 ​

现象：某材质在移动端阴影深度 Pass 中采样了本不该出现的 SceneTexture（SceneColor / SceneDepth 等），导致崩溃。 本文先梳理「材质如何进入阴影 Pass」的完整机制，再定位崩溃的精确条件与逻辑链。

一、背景：阴影 Pass 通常不需要 PS ​

阴影深度 Pass 的产物只有「深度」，不关心颜色。对一个普通不透明物体，VS 算出顶点位置后，硬件光栅化阶段就自动得到深度，完全不需要 Pixel Shader（Null PS），能省一大笔开销。

只有当材质需要在「像素级别」做判断/丢弃时（镂空、半透裁剪等），阴影 Pass 才会挂上 PS。而崩溃，正是发生在「带 PS 的阴影 Pass」里。

二、一个 mesh / 材质如何进入阴影 Pass：四个阶段 ​

一个 mesh / 材质要不要进阴影 Pass、进去后用什么 shader，是下面四个阶段层层筛选的结果：

mermaid
flowchart TD
    A["① InitViews：物体与灯光是否有阴影关系"] --> B["② AddMeshBatch：四个 if 判断条件"]
    B --> C["③ 材质降级 Override：是否替换为默认材质"]
    C --> D["④ GetShadowDepthPassShaders：要不要挂 PS"]
    A -. 不满足 .-> X["不进阴影 Pass，结束"]
    B -. 不满足 .-> X

阶段 1：InitViews —— 物体与灯光的阴影关系（物体级） ​

判断物体和灯光有没有阴影关系、哪盏灯会对它投影。需要满足：

  1. 物体组件 Primitive 勾选了 Cast Shadow；
  2. 与灯光有交互，处于灯光的投影范围内；
  3. 没有被剔除。

这一层是纯物体级筛选，不是当前重点，不多赘述。

阶段 2：AddMeshBatch —— 四个 if 判断条件（物体级 + 材质级） ​

cpp
if (MeshBatch.CastShadow)                                          // 条件1：网格批次级标志
{
    ...
    const bool bShouldCastShadow = Material.ShouldCastDynamicShadows();  // 条件2：材质级，含 Cast Shadow / Cast Dynamic Shadow as Masked 等
    ...
    if ((bShouldCastShadow || (bReflectiveShadowmap && (...)))
        && ShouldIncludeDomainInMeshPass(Material.GetMaterialDomain())   // 条件3：Domain 不能是 MD_Volume（体积材质走别的 pass）
        && ShouldIncludeMaterialInDefaultOpaquePass(Material))           // 条件4：排除 Sky 和 SingleLayerWater
    {...}
    ...
}
条件判断含义决定层级
1MeshBatch.CastShadow网格批次允许投影（来自组件/Primitive）物体级
2Material.ShouldCastDynamicShadows()材质本身是否投动态阴影（blend 模式 + Cast-as-Masked 等设置）材质级
3ShouldIncludeDomainInMeshPassDomain 不能是 MD_Volume材质级
4ShouldIncludeMaterialInDefaultOpaquePass排除 Sky / SingleLayerWater材质级

阶段 3：材质降级 Override —— 普通不透明材质被替换为默认材质 ​

cpp
// OverrideWithDefaultMaterialForShadowDepth (:634)
if (InOutMaterialResource->WritesEveryPixel(true) &&                        // 不是 Masked
    !InOutMaterialResource->MaterialModifiesMeshPosition_RenderThread() &&  // 没用 WPO
    !bReflectiveShadowmap)                                                  // 不是 RSM
{
    const FMaterialRenderProxy* DefaultProxy = ... UMaterial::GetDefaultMaterial(MD_Surface)->GetRenderProxy();
    ...
    InOutMaterialRenderProxy = DefaultProxy;   // ← 直接换成「默认材质」
    InOutMaterialResource = DefaultMaterialResource;
}

如果材质是普通不透明、没用 WPO、不是 RSM，阴影里它只是个实心轮廓，无需复杂材质图，于是被直接替换为引擎默认材质。默认材质里没有任何场景纹理引用，因此后续编出的阴影 shader 干净，不会崩。

只有满足以下之一的材质，才会保留自己的材质图进入阴影渲染（也正是潜在崩溃人群）：

  1. Masked（WritesEveryPixel 为假）；
  2. 使用了 WPO（顶点要动，必须保留自己的 VS）；
  3. RSM（GI 路径）。

阶段 4：GetShadowDepthPassShaders —— 最终决定要不要挂 PS ​

cpp
// ShadowDepthRendering.cpp L769-L773
if (Material.WritesEveryPixel(true) && !bUsePerspectiveCorrectShadowDepths && !bRenderingReflectiveShadowMaps && VertexFactory->SupportsNullPixelShader())
{
    // No pixel shader necessary.
    PixelShader.Reset(); // VS-only
}

四个 if 条件的含义：

条件含义不满足 = 需要 PS 的情况
WritesEveryPixel(true)每个像素都铁定存在、不会被丢弃Masked 材质会丢弃像素（镂空）→ false → 必须要 PS
!bUsePerspectiveCorrectShadowDepths不需要「透视正确」的特殊阴影需要透视正确（spot/逐物体）时要 PS。本工程移动端默认关闭
!bRenderingReflectiveShadowMaps不是 RSM（反射阴影图，GI 用）是 RSM → 要 PS
SupportsNullPixelShader()该顶点工厂支持空 PS某些 VF 不支持 → 要 PS

三、是否需要 PS 的整体判断流程 ​

mermaid
flowchart TD
    A["物体投阴影吗？"] -- 否 --> Z["不进阴影 Pass，结束"]
    A -- 是 --> B["进入 Shadow Depth Pass"]
    B --> C{"材质能写满每个像素吗？<br/>(不透明 / 不裁剪)"}
    C -- 是 --> D["省掉 PS（Null PS）<br/>只用 VS 写深度 ✅最省"]
    C -- "否（Masked / Cast-as-Masked / 透视正确 / RSM）" --> E["必须挂 PS<br/>在 PS 里做 clip 裁剪"]

一般只有在镂空或半透材质做阴影时，才需要在深度 Pass 挂上 PS 来做 mask 裁剪，从而得到阴影部分镂空的效果。

先只看「镂空」这条触发路径，透视正确与 RSM 的情况后续再补充。

四、崩溃根因 ​

崩溃的主要原因：在带 PS 的深度 Pass 中，采样到了本不应被采样的 SceneTexture（SceneColor / SceneDepth 等）。

正常流程下阴影 Pass 本就不该用到 SceneTexture，但这个操作没有被直接禁止，于是埋下隐患。

4.1 为什么只有连在 Opacity 上才崩 ​

阴影 PS 在移动端（非 RSM）真正消费的材质输出只有 Opacity / OpacityMask（被 clip() 使用）：

  • 场景纹理连到 Opacity → 采样被 clip() 用到 → 保留 → 引用 SceneTexture → 崩；
  • 场景纹理连到 BaseColor / Emissive → 阴影 PS 中无人读取 → 被编译器死代码消除 → 不引用 SceneTexture → 不崩。

RSM 路径会读取 BaseColor / Emissive，那种情况下连接到这些属性也可能出问题。此处先 mark，后续分析 RSM 时再补充。

五、崩溃逻辑链 ​

mermaid
flowchart LR
    A["开启 Cast-as-Masked<br/>→ 深度 Pass 带上 PS"] --> B["材质节点采样 SceneTexture<br/>(如 DepthFade 内部采样 SceneDepth，<br/>或 SceneTexture 节点)"]
    B --> C["采样结果连到 Opacity<br/>(阴影 Pass 唯一被关注的属性)"]
    C --> D["阴影阶段引用未就绪的 SceneTexture<br/>→ 崩溃"]
  1. 开启 Cast-as-Masked：让这个深度 Pass 带上 PS，从而产生采样其它纹理的可能；
  2. 节点采样 SceneTexture：DepthFade 节点内部会采样 SceneDepth，这是直接导致崩溃的诱因。除它之外，一些自定义节点（如 SceneTexture 节点）也可能采样到 SceneTexture；
  3. 连到 Opacity：因为 Opacity 是阴影 Pass 中唯一被关注的属性，采样会被保留进 PS，最终在阴影阶段引用未就绪的 SceneTexture，导致崩溃。

七、修复方法 ​

崩溃的核心原因是：移动端 ShadowDepthPass 阶段没有有效的 SceneTexture，但材质编译出的阴影 PS 仍然保留了对 SceneTexture 的采样，最终访问到未就绪的 MobileSceneTextures。

因此修复思路不是继续尝试给阴影 Pass 绑定一份全局 MobileSceneTextures，而是在阴影 PS 编译阶段直接把 SceneTexture 读取折叠成常量/dummy 值，让最终 shader 不再依赖真实的场景纹理资源。

7.1 在 ShadowDepth PS 编译环境中启用 SCENE_TEXTURES_DISABLED ​

UE 已经提供了 SCENE_TEXTURES_DISABLED 宏，用于在不允许访问 SceneTexture 的 pass 中，将相关读取替换为默认值。因此第一步是在移动端、非 RSM 的 ShadowDepth PS 编译环境里打开这个宏：

cpp
// Engine\Source\Runtime\Renderer\Private\ShadowDepthRendering.cpp L591-L598
static void ModifyCompilationEnvironment(const FMaterialShaderPermutationParameters& Parameters, FShaderCompilerEnvironment& OutEnvironment)
{
	FShadowDepthBasePS::ModifyCompilationEnvironment(Parameters, OutEnvironment);
	OutEnvironment.SetDefine(TEXT("PERSPECTIVE_CORRECT_DEPTH"), (uint32)(ShaderMode == PixelShadowDepth_PerspectiveCorrect));
	OutEnvironment.SetDefine(TEXT("ONEPASS_POINTLIGHT_SHADOW"), (uint32)(ShaderMode == PixelShadowDepth_OnePassPointLight));
	OutEnvironment.SetDefine(TEXT("REFLECTIVE_SHADOW_MAP"), (uint32)bRenderReflectiveShadowMap);

	// Mobile shadow depth PS does not have valid scene textures; fold scene texture reads to constants
	// to avoid referencing the not-yet-ready MobileSceneTextures (same as prepass / translucent shadow depth).
	if (IsMobilePlatform(Parameters.Platform) && !bRenderReflectiveShadowMap)
	{
		OutEnvironment.SetDefine(TEXT("SCENE_TEXTURES_DISABLED"), 1u);
	}
}

这一步可以覆盖 DepthFade / SceneDepth 这类通过内部 helper 访问 SceneDepth 的路径，避免它们在阴影 PS 中继续引用真实的 MobileSceneTextures。

7.2 补齐 MobileSceneTextureLookup 对 SCENE_TEXTURES_DISABLED 的处理 ​

仅打开 SCENE_TEXTURES_DISABLED 还不够：当前版本的 MobileSceneTextureLookup 没有处理这个宏，因此直接使用 SceneTexture 节点时，仍然可能生成真实的 SceneTexture 采样。

UE5 已经补上了这部分逻辑，可以迁移对应实现：当 SCENE_TEXTURES_DISABLED 为 1 时，不再访问真实纹理，而是按 dummy scene texture 的默认语义返回常量值。

hlsl
// Engine\Shaders\Private\MaterialTemplate.ush L2047
MaterialFloat4 MobileSceneTextureLookup(inout FMaterialPixelParameters Parameters, int SceneTextureId, float2 UV)
{
#if SCENE_TEXTURES_DISABLED
	// When scene textures are disabled, the output is matched to the dummy scene texture defaults.
	switch(SceneTextureId)
	{
	case PPI_SceneDepth:
	case PPI_CustomDepth:
		return ConvertFromDeviceZ(0.0f);
	case PPI_MaterialAO:
	case PPI_CustomStencil:
		return 1.0f;
	default:
		return 0.0f;
	}
#endif
...
}

7.3 修复效果 ​

完成以上两处修改后，移动端 ShadowDepthPass 中即使材质的 Opacity 路径引用了 DepthFade / SceneDepth / SceneTexture，最终 shader 也会走常量返回路径，而不会再绑定或采样真实的 MobileSceneTextures。这样可以从根上避开 shadow depth 阶段访问未就绪 SceneTexture 导致的崩溃。
