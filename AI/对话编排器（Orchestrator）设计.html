对话编排器（Orchestrator）设计文档 ​

类型：架构设计说明（自包含，面向任何读者） 阅读前提：无。本文不假设读者了解任何特定系统、框架或代码库。 示例场景：为便于理解，全文以一个「个人事务助理」类应用举例（记账、提醒、报表等）。这些只是说明性示例，编排器本身与具体业务领域无关。

0. 这是什么 ​

对话编排器是一个处理"自然语言请求"的引擎：

一条用户消息 ──▶ [ 编排器 ] ──▶ 一条回复

它的核心主张是 "程序主导、AI 辅助"：

  • AI（大语言模型，LLM）只做两件事：把用户的话解析成结构化的意图与参数、撰写自由文字（例如报表叙述）。
  • 定位、算数、校验、状态管理、执行、撤销，全部由确定性程序负责。

换句话说：凡是能用确定性代码解决的，绝不交给 LLM 猜。 只有在真正需要"语义理解"或"自由创作"时才调用 LLM。

为什么这么设计 ​

传统做法是让 LLM 主导整个循环——它自己决定调用哪个功能、传什么参数、调用几次，程序只在事后兜底。这带来三个问题：

  1. 易错且难纠：金额、分类、日期全靠 LLM 从自由文本里猜，错了往往已经落库。
  2. 不可控、不可测：多步任务靠 LLM 在循环里自由发挥，数字可能是它自己算的（幻觉），链路不可预测、延迟不定。
  3. 难维护：行为依赖冗长的提示词，靠"提示词祈祷"，难回归。

编排器把这套翻转过来：程序掌舵，AI 打下手。

1. 职责边界：做什么，不做什么 ​

编排器有清晰的边界。理解这条边界是理解整份设计的关键。

维度编排器负责不负责（由外部提供）
传输—消息的收发与渲染（消息通道）
并发假定同一会话串行到达同会话请求的串行保护
理解意图分类、参数抽取、指代消解LLM 供应商本身
调度分层路由、粘性状态、回退策略—
执行何时调、调哪个、传什么、幂等、可撤销业务动作的具体实现
状态会话上下文（焦点/撤销栈/草稿/近期实体）的逻辑状态的物理存储
数据—业务数据的持久化

一句话：编排器拥有"怎么用工具"的决策权，但不拥有"工具本身"，也不拥有"消息通道"。

这种解耦带来三个好处：

  • 可移植：更换消息通道、LLM 供应商或存储实现，编排器核心零改动。
  • 可测试：核心是纯逻辑，依赖全是可替换的抽象接口 → 可用表驱动单测覆盖。
  • 可演进：高频的确定性场景可逐步从"AI 兜底"沉淀为"确定性技能"，互不影响。

2. 架构总览：端口与适配器 ​

编排器采用端口与适配器（Ports & Adapters，又称六边形架构）：核心只依赖一组抽象**端口（Port）接口，外部系统通过适配器（Adapter）**把真实实现注入进来。核心代码里不出现任何具体的通道、框架或数据库。

2.1 输入 / 输出契约 ​

编排器只认识与通道无关的请求与回复，通道细节全部留给适配器：

ts
interface OrchestratorRequest {
  channelId: string // 会话隔离键（同一会话的消息共享上下文）
  userId: string // 业务用户标识（用于数据归属与鉴权）
  text: string // 用户原文
  requestId: string // 本次请求的唯一标识，用于幂等去重
}

interface OrchestratorReply {
  text: string // 主体文案（具体样式由渲染器决定）
  card?: EchoCard | DraftCard | ListCard // 可选：结构化卡片
  actionsHint?: string[] // 可选：可用后续操作，如 ["撤销", "全部撤销"]
}

// 唯一入口
interface Orchestrator {
  handle(req: OrchestratorRequest): Promise<OrchestratorReply>
}

2.2 依赖端口清单（接入方必须提供） ​

下列签名是把编排器接入任何系统的最小契约。命名为 Port，其实现称为 Adapter。

ts
// 1) 工具端口：业务动作的执行与查询
interface ToolPort {
  get(intent: string): {
    effectLevel: 'read' | 'write' | 'delete'
    execute(userId: string, params: object): Promise<unknown>
  }
  // 目标定位/撤销所需的确定性查询与快照
  queryEntities(
    userId: string,
    domain: string,
    sel: TargetSelector,
  ): Promise<Entity[]>
  snapshot(userId: string, domain: string, id: string): Promise<object> // 执行前旧值
}

// 2) LLM 端口：仅结构化输出、temperature=0（消除随机性）
interface LlmPort {
  classify(input: DispatcherInput): Promise<Route> // 意图+抽参+多意图+缺字段，一次完成
  writeSections(
    facts: object,
    sections: string[],
    tone?: string,
  ): Promise<Record<string, string>>
  classifyEdit(text: string, sections: object): Promise<EditKind>
}

// 3) 状态端口：会话上下文的物理存储
interface StatePort {
  recentEntities: KV<RecentEntities> // 序号→id / 近期操作
  activeForm: KV<ActiveForm> // 单条多轮补全焦点
  undoStack: UndoStore // 撤销栈（内存 + 落库）
  reportDraft: KV<ReportDraft> // 报表草稿
}

// 4) 幂等端口：写操作去重
interface LedgerPort {
  seen(key: string): Promise<boolean>
  remember(key: string, result: unknown, before?: object): Promise<void> // before 供撤销重建
}

// 5) 时钟端口：TTL/过期判定可注入，便于测试
interface ClockPort {
  now(): number
}

编排器对 LLM 的唯一信任是"语义理解"——分类、抽参、写文字。定位、算数、校验、状态、执行、撤销，一律由程序负责。

3. 内部结构与主流程 ​

编排器内部由五个协作单元构成：

  • Dispatcher（主流程）：请求的总入口，按优先级依次尝试各处理门。
  • Router / StickyGate（调度）：决定一条消息该走哪条路径。
  • TierExecutor（分层执行）：把决策分派到对应的执行层。
  • Skills（确定性技能）：具体的定位、写入、报表、多任务等能力。
  • StateStore（状态逻辑）：管理会话上下文。

3.1 主流程：顺序即优先级 ​

核心原则：粘性状态永远优先于路由。 当用户正处在一个"未完成的上下文"里（例如助理刚问"记多少钱？"，用户回"35"），这个"35"应当直接补全，而不是被重新当成一条新指令去分类。

三种"粘性上下文"各有独立判定门，按下列顺序尝试；每轮至多调用一次 LLM 分类：

要点：

  • 粘性状态优先于路由：焦点在等金额时，"35" 直接补槽。
  • 每轮至多一次 LLM 分类：焦点判不准时，复用同一次分类结果来判断"续填还是换主题"，绝不重复调用。
  • 置信度必须参与决策：分类置信度低于阈值，或 LLM 调用失败，则回退到"追问澄清"或"兜底层"，绝不带着不确定硬执行写操作。这是"不确定就向更稳处回退"原则的落地点。

3.2 执行引擎：分层技能 + 统一写出口 ​

分类结果交给 TierExecutor，按意图分派到相应层；所有写操作统一经过 WriteExecutor，从而保证一致的幂等、回显与"可撤销"。

4. 分层执行模型（Tier） ​

编排器把处理路径按"稳 → 自由"分层：越稳的路径越确定、越可测、越可撤销；越自由的路径护栏越重。 一条请求命中哪一层，取决于它有多明确、是否有依赖、是否有预定义流程。

层名称何时用LLM 参与可撤销
L0规则 / 粘性确认词、序号、撤销、焦点续填无—
L1直接工具单个功能、参数明确的读操作仅抽参—
L2确定性技能单条增/查/改/删仅抽 {目标, 变更}✅
L2+独立多意图无依赖的组合（"记一笔 + 提个醒"）仅拆队列✅ 逐条
L3报表流程汇总、周期性报告仅写文字定稿前不落库
L3.5依赖型计划后一步用到前一步的结果拆结构化计划✅
L4受约束智能体无预定义流程的杂项（兜底逃生舱）自由（护栏内）尽量

回退方向永远朝"更稳"：不确定时 → 追问 / 列表选号 / 降级到兜底层。任何未被确定性层覆盖的意图都回退到 L4，保证"任何时候都不比纯 AI 方案差"。随着确定性技能的完善，L4 的占比应逐步下降。

5. 状态模型：编排器拥有的上下文 ​

编排器不依赖 LLM 的聊天历史来"回忆"上下文，而是用程序显式管理四类状态。所有状态按 channelId（会话）隔离，通过 StatePort 存取——逻辑归编排器，物理存储归适配器。

状态存什么生命周期作用
近期实体序号→id、刚操作过的记录30 分钟指代消解（"第 2 条""刚才那笔"）
活动焦点意图 + 目标 + 已抽字段 + 还缺什么5 分钟 / 换主题即弃单条记录的多轮补全
撤销栈操作前/后的补偿快照5 分钟 / 限深 5 / 按请求分组一键撤销（乐观写的纠错网）
报表草稿事实包 + 章节文字~30 分钟 / 显式清报表的多轮编辑

几个关键设计：

  • 撤销栈持久化：操作前的快照除进内存外同步落库，进程重启后可按 (channelId, requestId, 时间窗口) 重建栈顶——即使服务刚重启，用户对"刚才那笔"依然可以撤销。
  • 撤销触发：出现"撤销 / 撤回 / 恢复"等词即弹出栈顶并执行反向操作（撤增=删、撤改=写回旧值、撤删=用快照重建）；连说多次可逐层回退。"全部撤销"按同一请求分组弹出，避免一次多写把上一条挤出栈后无法整体撤销。
  • 报表草稿宽容：中途插入别的查询不会丢草稿，仅在显式定稿/弃稿或过期后才清。
  • 状态可丢：活动焦点、报表草稿等短时上下文重启丢失可接受；目标定位以数据检索为主（见 §6.1），具体描述在重启后仍可定位。

6. 核心技能详解 ​

6.1 目标定位（TargetResolver，增查改删共用） ​

核心洞察：目标定位本质是"检索"，而不是"回放记忆"。 用户描述得越具体，越不依赖上下文；会话记忆只是加速指代的手段，而非唯一来源。

用户对"哪条记录"的描述被抽取为一个 TargetSelector（序号、时间近度、语义文本、金额、日期范围等），解析算法是"构造候选集 → 多约束过滤 → 计数"，得到三种结果：

构造候选集：
  有序号 → 从序号缓存取（最精确）
  说"刚刚" → 先看近期操作过的，未命中再查库
  否则 → 按文本 + 日期 + 金额查库
再用全部约束过滤，然后计数：
  命中 1 条  → locked     锁定，继续执行
  命中 0 条  → none       列出最近几条 + 转焦点(缺目标)
  命中 N 条  → ambiguous  列表 + 序号让用户选（退回最稳的序号机制）

安全差异：删除操作要求 locked（唯一命中）才乐观执行；只要不唯一，一律退回列表选号——因为删错的代价与改错不对称（改错能回显纠正，删错是先删了才被发现）。

6.2 乐观写统一出口（WriteExecutor） ​

所有写操作（增/改/删）都经过这里，保证一致的行为：

ts
async execute(req, task): Promise<OrchestratorReply> {
  const key = idemKey(req, task)
  if (await ledger.seen(key)) return ledger.result(key)        // 1) 幂等：重复请求直接返回旧结果
  const before = task.op !== 'create'                          // 2) 执行前抓旧值快照
      ? await tool.snapshot(req.userId, task.domain, task.targetId) : undefined
  const res = await tool.get(task.intent).execute(req.userId, task.params) // 3) 执行
  await ledger.remember(key, res, before)
  undoStack.push(req.channelId, { requestId: req.requestId, op: task.op, before, after: task.params, label })
  return renderEchoCard(task, res, before)                     // 4) 回显 "¥35 → ¥50"
}

"乐观写 + 回显 + 可撤销"是本设计对写操作的核心策略：不在写前反复确认（打断体验），而是先执行、把结果清晰回显给用户，并始终提供一键撤销。回显卡片本身就是纠错网。

6.3 报表流程（ReportWorkflow） ​

一句话原则：数字来自程序，文字来自 LLM。

生成：
  gatherFacts   —— 程序批量查询与计算，所有数字（含派生量）都在这里产生
  writeSections —— LLM 只负责把事实组织成文字，不做任何计算
  validateNumbers —— 校验文字里的每个数字都能在事实包中找到
  存草稿
编辑（草稿存在时）：
  改数据  → 重新取数 + 重写受影响章节 + 再校验
  改文字  → 只重写文字 + 校验数字未被篡改
  定稿    → 需要落库的报表落库；纯展示类不落库
  弃稿    → 丢弃草稿

数字校验（报表命根）：从生成的文字中抽取金额/统计类 token，逐一在事实包里匹配（允许格式差异，如 ¥35 / 35元）。为避免误伤：

  • 事实包须包含派生量：环比、日均、占比、合计、排名等由程序预先算好放进事实包——因为叙述天然会引用这些派生值，若事实包只有原始数字，会把正确的派生值误判为幻觉。
  • 按语境过滤 token：日期（"7月"）、序号（"第2条"）等非金额数字不参与校验。
  • 失败处理：不匹配 → 提示 LLM 重写一次；仍不匹配 → 该数字降级为"见明细"，绝不呈现错误数字。

6.4 独立多意图（CompoundRunner） ​

把一句话拆成有界任务队列（例如最多 3 个已知技能），各自独立地乐观执行、各自回显、各自可撤销。

  • 执行前先做独立性校验：若任务之间存在依赖（"删掉 A，然后基于剩下的做 B"）→ 升级到依赖型计划（§6.5），不并行。
  • 部分失败各自独立：成功的照做、缺信息的转补全、硬失败的明确报错，不回滚已成功的。
  • 单焦点约束：若多个任务都缺信息，只把第一个转为"活动焦点"追问，其余在汇总里提示"待补充"。

6.5 依赖型计划（PlanExecutor，重护栏） ​

用于"后一步用到前一步结果"的场景。关键：中间数据由程序解析引用来传递，绝不经过 LLM。

计划是一串带引用的步骤，例如：

step1: 统计上一周期某类别的合计
step2: 把某项设置为 "$1.total"   ← 引用 step1 的输出

执行器逐步运行，遇到 $1.total 这类引用时由程序替换成真实值（而非让 LLM 转述数字，避免二次幻觉）。护栏：步数上限、每步必须是已知技能、引用必须能解析、参数类型校验、含删除步则执行前暂停确认；非法计划不硬跑，回退追问或转兜底层。

6.6 受约束智能体（兜底逃生舱，L4） ​

当前面所有确定性层都不匹配时，才启用一个受约束的自由智能体兜底：限制最大轮数与总超时、写操作仍走统一出口（因而仍可撤销）、保留最终回复的防幻觉校验、全程记录日志。它是永久保留的安全阀——同时，日志用于发现高频模式，再把这些模式"编译"成更稳的确定性技能。

7. LLM 的调用契约 ​

编排器对 LLM 的所有调用都要求结构化输出（如 JSON），并设 temperature=0 以消除随机性。共四类：

调用输入输出
分类抽取用户消息 + 最近 1~2 轮 + 意图清单{ tasks[], plan?, confidence }；针对已有记录含 {目标, 变更}
补槽抽取单句 + 当前焦点{ 变更 } 或 { 目标 }（上者子集）
报表写作事实包 + 章节列表 + 语气{ 章节名: 文字 }；只能引用事实包中的数字，不得计算或臆造
报表编辑分类编辑指令 + 当前章节{ 类型: 改数据/改文字/定稿/弃稿, 目标章节? }

8. 可靠性与校验 ​

编排器的可靠性建立在四道确定性防线上：

  • 幂等：所有写操作按 (channelId, userId, requestId, intent, 参数哈希) 去重；同一请求重复到达只执行一次。多意图/计划按子项分别记账。
  • 数字校验：报表里的数字必须来自程序算出的事实包（见 §6.3）。
  • 执行证据校验（防幻觉）：任何"已记录 / 已修改 / 已删除"的措辞，必须有对应的成功写执行记录，否则拦截——杜绝"嘴上说做了、实际没做"。
  • 撤销一致性：旧值快照在执行前抓取；撤销严格按栈序反向执行，多次修改可逐层回退（35→50→80，撤两次回到 35）。
  • 越权隔离：会话级缓存按 channelId 隔离、数据查询按 userId 归属；入口处校验两者绑定，防止跨用户指代到他人记录。

9. 运行示例（消息 → 内部走查 → 回复） ​

下面用连续对话演示编排器如何在各层之间流转。▶ 表示编排器内部步骤，💬 表示最终回复。

例 1 · 单条记录（L2 直落） ​

用户：午饭 35
▶ L0 三道门均未命中（无撤销词/草稿/焦点）
▶ 路由：规则识别出金额 → 意图=新增支出，置信度高
▶ 程序确定性解析金额=35、日期=今天；LLM 语义匹配类别=餐饮
▶ 乐观写：幂等未命中 → 执行 → 压撤销栈
💬 已记：支出 · 餐饮 · ¥35 · 今天　（回复"撤销"可撤回）

例 2 · 多轮补全（L0 焦点续填，不重复分类） ​

用户：记一笔
▶ 路由：意图=新增支出，但缺金额 → 建立"活动焦点(缺字段)"
💬 记多少钱呀？

用户：35
▶ 焦点存在；规则判定"35"像金额 → 直接续填（不再调用 LLM）
▶ 补全金额=35 → 齐 → 乐观写
💬 已记：支出 · 餐饮 · ¥35 · 今天　（回复"撤销"可撤回）

例 3 · 指代 + 修改（L2，回显即纠错） ​

用户：把刚才那笔咖啡改成 50
▶ 路由：意图=修改，目标={时间:刚刚, 文本:"咖啡"}，变更={金额:50}
▶ 目标定位：先查近期操作 → 唯一命中 → locked
▶ 乐观写：旧值={金额:35} → 执行 → 新值={金额:50} → 压撤销栈
💬 已改：餐饮 · ¥35 → ¥50　（回复"撤销"可还原）

例 4 · 撤销（L0 粘性触发） ​

用户：撤销
▶ 撤销栈非空且命中撤销词 → 撤销处理
▶ 弹出栈顶(修改, 旧值={金额:35}) → 反向：写回 35
💬 已撤销：餐饮 ¥50 → ¥35

例 5 · 焦点中的话题切换（复用同一次分类） ​

（此时"活动焦点"正在等一笔支出的金额）
用户：查一下这个月的账单
▶ 焦点存在；规则判定：不像金额/序号/确认 → 需要分类
▶ 路由分类一次 → 意图=查询账单（与当前焦点主题不同）
▶ 用这次分类结果判定 → 换主题：清掉焦点，复用同一结果继续执行（未二次调用 LLM）
💬 本月共 12 笔，合计 ¥1,860 …（列表；写序号缓存，供后续"删第 2 条"指代）

例 6 · 报表（L3：事实包 + 草稿 + 校验 + 编辑） ​

用户：生成上月消费汇总
▶ 报表流程 · 生成
▶ 取数：程序统计各类别金额，并预先算好派生量（环比 +12%、日均 ¥62）
▶ LLM 只写"总览/消费分析/环比/建议"四段文字
▶ 校验：文中 ¥1,860 / +12% / ¥62 均能在事实包中命中 ✅ → 存草稿
💬 【上月消费汇总·草稿】总览：… 环比 +12% …（可继续编辑，或说"存为月报"）

用户：语气正式点，结论那段删掉
▶ 草稿存在且像编辑指令 → 编辑处理
▶ 编辑类型=只改文字 → 重写文字 → 再校验数字未被篡改 ✅
💬 （更正式、无结论段的草稿）

例 7 · 独立多意图（L2+） ​

用户：记一笔 35 餐饮，顺便提醒我明天 9 点交房租
▶ 路由：拆成两个任务[新增支出{35,餐饮}, 新增提醒{明天9点,交房租}]，无相互引用
▶ 独立性校验通过 → 各自乐观写，各自压撤销栈
💬 ✅ 已记：餐饮 ¥35　✅ 已提醒：明天 09:00 交房租　（回复"全部撤销"可一起撤）

例 8 · 依赖型计划（L3.5，程序传引用） ​

用户：把上月餐饮总额设成这个月的餐饮预算
▶ 路由产出结构化计划：
    step1 统计{上月, 餐饮}
    step2 设置预算{类别:餐饮, 金额:"$1.total"}   ← 引用
▶ 计划校验通过 → 执行
    step1 → 输出 {total: 1860}
    解析引用：$1.total → 1860（由程序替换，不经 LLM）
    step2 → 设置预算{金额:1860} → 压撤销栈
💬 已设置：餐饮 本月预算 ¥1,860（来自上月实际支出）

▶ 路由：无匹配的确定性技能；需要多步自由推理

10. 可测试性与可观测性 ​

因为核心只依赖抽象端口，每个单元都能独立地表驱动测试：

单元输入 → 期望
路由消息 → 期望的 Route（含多意图/目标选择器）
目标定位给定缓存/近期操作/数据 → 期望 locked/none/ambiguous
乐观写 + 撤销栈执行→压栈→撤销→状态还原；多次改多次撤
报表流程固定事实包 → 生成/编辑 → 校验必过；注入错误数字→被拦
多意图部分失败 → 成功项不回滚、失败项报错/转补全
依赖型计划引用解析、非法计划回退、删除步暂停确认
粘性判定等金额时"35"→续填；"查账单"→换主题
置信度闸低置信 → 进入追问/兜底，不硬执行写

可观测性（覆盖全层，而非仅兜底层）：埋点各层的命中分布、每轮 LLM 调用次数（应恒 ≤1）、置信度回退率；并以撤销率、报表数字校验失败率作为"写正确性/报表正确性"的核心指标——它们是判断"某个高频意图能否从兜底层沉淀为确定性技能"的量化依据。

11. 把编排器接入一个新系统 ​

  1. 实现 5 个端口适配器：工具端口（含查询与快照）、LLM 端口、状态端口、幂等端口、时钟端口。
  2. 在消息入站处构造 OrchestratorRequest，并在会话粒度的串行保护下调用 orchestrator.handle(req)。
  3. 把 OrchestratorReply 渲染为该通道的展示形式（富文本 / 卡片 / 纯文本）。
  4. 注册业务工具到工具端口（意图 → { 副作用级别, 执行函数 }），编排器即可调度它们。
  5. 配置阈值：置信度回退阈值、各状态的 TTL、撤销栈的深度与窗口、兜底层的轮数与超时。

只要满足这五步，编排器核心就能在任意消息通道、任意业务领域、任意技术栈上运行，且与通道代码完全解耦。

12. 设计原则速查 ​

  • 程序主导，AI 辅助：能确定性解决的绝不交给 LLM。
  • 数字来自代码，文字来自 LLM：任何金额/统计只出自程序，LLM 只组织语言。
  • 稳 → 自由分层：越稳越确定、可测、可撤销；不确定就向更稳处回退。
  • 乐观写 + 回显 + 可撤销：不打断体验，用回显和一键撤销兜错。
  • 状态由程序显式拥有：指代、补全、撤销、草稿都是显式状态，不靠 LLM 从历史里"悟"。
  • 端口与适配器：核心零框架依赖，可移植、可测试、可演进。
